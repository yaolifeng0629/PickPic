# Design: Version Unify & GitHub Entry Toggle

Date: 2026-05-13
Status: Draft

---

## 1. Project Context

This is a **Plasmo Chrome extension** (React 18 + TypeScript + Tailwind CSS). State management uses Zustand (`store/*.ts`).

## 2. Problem Statement

- `package.json` has a version mismatch: root `version` is `0.0.1` while `manifest.version` is already `2.0.0`.
- Two UI sections reference GitHub (Issues link and "Star on GitHub" in Settings), but there is no way to hide them at runtime.

## 3. Goals

1. Unify all version fields to `2.0.0`.
2. Make GitHub entry visibility **configurable and persistent** across sessions.

## 4. Non-Goals

- Do NOT remove i18n translation keys for GitHub-related text (keep for future reuse).
- Do NOT add a UI settings page beyond the single toggle.
- Live cross-device sync of the toggle state is not required.

## 5. Design

### 5.1 Version Unification

| File | Field | Current | Target |
|------|-------|---------|--------|
| `package.json` | `version` | `0.0.1` | `2.0.0` |
| `package.json` | `manifest.version` | `2.0.0` | `2.0.0` (no change) |

The root `version` field is what `npm` and `plasmo package` read; it must match the manifest version for consistency. `build/` manifests are generated by Plasmo and will inherit the correct version automatically on the next build.

### 5.2 GitHub Entry Toggle — Architecture

**Chosen approach: Chrome Storage + Zustand (Store A)**

A new `store/settings-store.ts` exports `useSettingsStore` (matching existing `useImageStore`, `useUIStore` conventions). It uses `zustand` for reactive UI state and `chrome.storage.sync` for persistence.

```
useSettingsStore (zustand)
  ├── showGithubEntry: boolean  (reactive state)
  ├── setShowGithubEntry(v)     (action: updates zustand + writes to chrome.storage.sync)
  └── init()                    (hydrate from chrome.storage.sync on mount)
```

**Data flow:**

1. On app mount (in `sidepanel.tsx`, the primary entry point), `useSettingsStore.getState().init()` is called once. It reads `chrome.storage.sync.get('settings')` and hydrates the store. A hydration flag prevents duplicate reads. This ensures persisted state is available before the user navigates to Settings.
2. User toggles the switch → `setShowGithubEntry(v)` updates zustand state AND writes back to `chrome.storage.sync`. If the write fails, zustand state is reverted to the previous value to keep UI and storage consistent.
3. `FeedbackTab` and `DonateTab` subscribe to `showGithubEntry` and conditionally render GitHub DOM.

### 5.3 UI Changes

All referenced files exist in the current codebase:
- `tabs/settings.tsx` — settings page entry
- `components/settings/feedback-tab.tsx` — exists
- `components/settings/donate-tab.tsx` — exists
- `locales/en/translation.json` and `locales/zh-CN/translation.json` — exist

**New component:** `components/settings/github-toggle.tsx`
- A simple labeled switch using a native checkbox styled with Tailwind CSS.
- Label text: `t('settings.showGithubEntry')` (new i18n key).
- **Placement:** Rendered inside `tabs/settings.tsx`, above the tab content area (`<main>`), so it is visible regardless of which tab is active.

**Modified components:**
- `components/settings/feedback-tab.tsx`: Conditionally render the section whose heading reads `{t('settings.githubIssues')}` (the block containing the GitHub Issues link and description).
- `components/settings/donate-tab.tsx`: Conditionally render the first support-method card whose title reads `{t('settings.starGithub')}` (the block containing the star-on-GitHub description).

### 5.4 State Definition

```ts
// store/settings-store.ts
interface SettingsState {
  showGithubEntry: boolean
  _hydrated: boolean          // guards against duplicate init() calls
  setShowGithubEntry: (v: boolean) => Promise<void>
  init: () => Promise<void>
}

const STORAGE_KEY = 'settings'
const DEFAULT_SETTINGS = { showGithubEntry: true }
```

`init()` is idempotent: if `_hydrated` is already `true`, it returns immediately. `sidepanel.tsx` calls `useSettingsStore.getState().init()` once on mount.

### 5.5 Error Handling

- Detection: wrap `chrome.storage.sync.get/set` calls in `try/catch` and check `typeof chrome !== 'undefined' && chrome.storage`.
- If the API is unavailable (e.g., running in a non-extension test environment), fall back to memory-only state (zustand default).
- **Hydration read failure** is silent and safe: the default `showGithubEntry: true` maintains backward compatibility (existing users continue seeing GitHub links).
- **Write failure:** if `chrome.storage.sync.set` throws or rejects, revert the zustand state to the previous value so the UI never shows a persisted state that was not actually saved. No user-facing error is shown.
- **Quota:** `chrome.storage.sync` has a 102,400-byte total quota. Storing a single boolean under `STORAGE_KEY = 'settings'` is well within limits. If future settings expand the object significantly, migration to `chrome.storage.local` should be considered.

### 5.6 i18n Keys (to add)

```json
{
  "settings": {
    "showGithubEntry": "Show GitHub links",
    "showGithubEntryDesc": "Display GitHub issue and star links in settings"
  }
}
```

Add to `locales/en/translation.json` and `locales/zh-CN/translation.json`. These are the only two locale files in the project.

## 6. Testing

- Verify `package.json` root version reads `2.0.0`.
- **First install / no prior storage:** open settings → GitHub links are visible by default (backward compatibility).
- Toggle off → GitHub links disappear from both tabs.
- **Persistence:** close the side panel (or browser) and reopen → toggle state persists.
- Toggle on → GitHub links reappear.

**Automated tests (out of scope for this change):** Unit tests for `useSettingsStore` persistence logic are not required. Manual verification of the toggle and persistence is sufficient.
